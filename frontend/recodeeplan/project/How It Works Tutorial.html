Merge mode

When two agent worktrees produce PRs that conflict, Merge mode spins up a third agent whose only job is to resolve the conflict — reading both sides, understanding intent, and writing a clean merge commit. You review the resolution, not the conflict markers.
⚠ conflict detected on apps/dashboard/src/feature-flags.ts
PR #421 (dashboard-rust-port)  ×  PR #438 (flags-cleanup-sweep)
1
Conflict detected at merge time
Base branch moved. Both PRs touched feature-flags.ts. GitHub blocks the merge.
2
Merge agent auto-spawns
A fresh worktree agent/merge/pr-421-vs-438 is created from the target branch. Both PRs get cherry-picked, conflict markers exposed.
3
Agent reads both sides + intent
It loads the PR descriptions, the commit messages, and the surrounding code. Not a three-way textual merge — a semantic one.
4
Resolution written + tests re-run
Conflict markers removed, both features preserved. Test suite runs before the agent reports back.
5
You review the merge commit
You get a side-by-side: their change, your change, the agent's resolution — with a one-line rationale. Approve to merge, reject to retry.

Plan mode

Plan mode turns the agent into a strategist. It won't touch code until you approve the plan — great for risky refactors or when you want to see the blast radius first.
1
Toggle Plan on in the composer
The checkbox next to Plan turns purple. Access mode is forced to Read only while the plan is drafted.
2
Prompt as usual
e.g. "migrate dashboard reads to Rust". The agent reads the repo but writes nothing.
3
Review the proposed plan
You get an ordered list of phases, files that will change, and risks. No worktree created yet.
4
Approve phases selectively
Check the phases you're happy with. Uncheck the rest. Edit the summary inline.
5
Flip access to On‑Request
Now the agent creates a worktree and starts executing approved phases, asking permission before each write.
